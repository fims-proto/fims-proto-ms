# AI Development Guide for fims-proto-ms

This document guides AI-assisted code changes in the repository. Its purpose is to provide clear contracts, transaction templates, error handling conventions, and a PR checklist so AI-generated code stays safe, testable and reviewable.

## 1. Purpose
- Help AI produce code that fits the project's architecture and coding conventions.
- Provide reusable templates for transactions, error handling, and PR metadata.

## 2. Key contracts and interfaces
- DataSource (internal/common/datasource/datasource.go)
  - GetConnection(ctx context.Context) *gorm.DB
  - EnableTransaction(ctx context.Context, transactionalFn func(txCtx context.Context) error) error
  - Always use the DataSource interface for DB operations. Do NOT open raw DB connections in handlers.

- Repository update pattern
  - Use repository-level update patterns like:
    - UpdateVoucher(ctx, voucherId, updateFn func(*voucher.Voucher) (*voucher.Voucher, error))
    - ReportRepo.UpdateItem(ctx, itemId, updateFn func(*report.Item) (*report.Item, error))
  - The update callback must use domain value object (VO) methods for state changes (e.g. `v.Post()`, `v.Audit()`).

- Numbering service
  - Keep the interface: numberingService.GenerateIdentifier(ctx, periodId, voucherType) -> string

## 3. Transaction template (must use)
- All critical write flows (create voucher, post voucher, audit/review, initialize ledgers, update report item) must run inside:

```
repo.EnableTx(ctx, func(txCtx context.Context) error {
    // use repo methods only
    // call domain VO methods for state changes
    // return error to roll back
    return nil
})
```

- Do NOT manage transactions at the HTTP layer.

## 4. Error model and localization
- Use `internal/common/errors.NewSlugError(slug, args...)` for business errors.
- Handlers should return slug errors and let the common error middleware map them to localized messages.
- When adding a new slug, update `i18n/zh-CN.json`.

## 5. Report generator (high risk area)
- File: `internal/report/domain/generator/generator.go`
- Important rules:
  - Items use data sources: `Sum` or `Formulas`.
  - Formula rules: `Net`, `Debit`, `Credit`, `Transaction`.
  - `ledgersCache` is used to reduce repeated DB reads. Keep or replace cautiously.
- Any change to formula rules or aggregation must be covered by unit+integration tests.

## 6. Posting (ledger update) rules
- File: `internal/general_ledger/app/command/post_voucher.go`
- Steps:
  1. Mark voucher as posted via `v.Post(poster)`
  2. Build records for all line items and their superior accounts
  3. Merge identical accounts (sum debit/credit)
  4. Call `UpdateLedgersByPeriodAndAccountIds` and `UpdateAuxiliaryLedgersByPeriodAndAccountIds`
- Maintain the merge and batch update pattern to avoid multiple writes and inconsistencies.

## 7. Schema and migrations
- The project uses GORM `AutoMigrate` in repository `Migrate()` methods.
- If you change model fields:
  - Evaluate AutoMigrate impact on existing data
  - Prefer backward-compatible changes (new nullable columns with defaults)
  - Document manual migration steps in PR if necessary

## 8. Performance guidance
- DB connection pooling: configured in `internal/common/datasource/connector.go` (MaxOpenConns=25)
- Report generation: prefer caching and chunked computation for large reports
- API latency target: <= 3 seconds per request (design target)

## 9. Permissions (suggested pattern)
- Roles: admin, accountant, auditor, supervisor
- Suggestion: implement role checks as a middleware at HTTP layer (e.g. `RequireRole(roleAccountant)`), but leave final decision to human reviewer.
